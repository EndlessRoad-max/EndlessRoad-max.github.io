初识JavaWeb

  1. 初识 JavaWeb 学习笔记

    1. 核心本质与概念

    什么是 JavaWeb？

    • 本质： Java 在网页端（服务器端）的应用。
    • 角色： 它是整个 Web 架构中的“服务器端大脑”，负责接收来自客户端（如浏览器）的 HTTP 请求，执行复杂的业务逻辑（如操作数据库），并向客户端返回处理结果（HTML 页面或数据）。
    • 技术基础： 在 JavaSE 的基础上（尤其是 I/O 流、多线程、Socket 网络通信等技术），封装了处理 Web 请求的组件和包。
    • 归属： 它是 JavaEE（Jakarta EE）规范中的一个核心子集，主要聚焦于与 Web 开发最紧密相关的技术（如 Servlet、JSP、JDBC）。

    硬件与软件的视角

    初学者常对“服务器”感到困惑，需要区分硬件和软件：

    • 硬件服务器： 任何安装了服务器软件并连接到网络的计算机。普通的个人电脑只要运行了 Tomcat 等软件，就可以作为一台服务器。机房中的服务器只是性能更强、稳定性更高的计算机。
    • 软件服务器（如 Tomcat）： 本质上是一个一直在运行的程序（进程），监听特定端口（如 8080），等待并处理客户端的请求。
    • 客户端： 如浏览器或手机 App，其主要职责是发起请求和渲染响应，将后端返回的数据可视化展示给用户。

    2. 核心组件探秘：Tomcat 服务器

    Tomcat 是由 Java 编写的程序，在 JavaWeb 开发中扮演着至关重要的角色，它可以被看作一家“全能型餐厅”，兼具两个核心身份：

    身份一：Web 服务器 (Web Server) —— “前台接待”

    主要负责处理网络连接和 HTTP 协议解析。

    • 建立连接： 底层监听特定端口（如 8080），接收客户端的 TCP/IP 请求。
    • 解析协议： 将复杂的 HTTP 请求报文字符串拆解，提取 URL、请求头、请求体等关键信息。
    • 对象封装： 将提取出的数据封装为 Java 对象（HttpServletRequest），并准备一个空的对象（HttpServletResponse）用于承载返回结果。
    • 静态资源处理： 如果客户端请求的是 HTML、CSS、图片等静态文件，Web 服务器会直接从磁盘读取并响应，无需执行 Java 业务代码。

    身份二：Servlet 容器 (Container) —— “后厨主管”

    负责管理和运行我们编写的 Java 业务代码（Servlet）。

    • 路由分发： 根据请求的 URL 路径，决定调用哪个具体的 Servlet 来处理。
    • 生命周期管理（控制反转）： 这是容器的核心功能。开发者不需要 new 一个 Servlet，也不写 main 方法去调用。Tomcat 负责 Servlet 的实例化、调用其核心方法处理请求，并在服务停止时销毁它。
    • 状态管理： 提供运行上下文，管理诸如 Session（用户会话状态）和 Application（全局共享数据）等资源。

    一次动态请求的完整生命周期

    1. 发起请求： 浏览器向服务器（例如 http://localhost:8080/login）发送 HTTP 请求。
    2. 接收与解析： Tomcat 的 Web 服务器模块接收网络字节流，解析为 HTTP 格式，并封装成 Request 和 Response 对象。
    3. 转交容器： Web 服务器将这两个对象交给 Servlet 容器。
    4. 路由与执行： 容器根据路径（/login）找到对应的 LoginServlet，并调用其处理方法（如 doGet 或 doPost）。
    5. 业务处理： LoginServlet 读取 Request 中的数据，执行业务逻辑（如数据库查询），并将结果写入 Response 对象。
    6. 返回结果： 容器将填充完毕的 Response 对象交还给 Web 服务器。Web 服务器将其转换为标准的 HTTP 响应报文，发送回浏览器。

    3. 核心秘密：文件到底放哪里？

    Web 项目的代码和资源并不是随便乱放的，必须遵循 Tomcat 的“美食广场”规则，统一存放在 Tomcat 安装目录下的 webapps 文件夹中（或通过虚拟路径映射）。在一个具体的项目文件夹（如 myApp）中，结构如下：

    A. 静态资源区 (公开区 / 餐厅前厅)

    • 位置： 直接放在项目根目录下（例如存放 index.html，或者建立 css/、img/ 文件夹存放对应资源）。
    • 特点： 浏览器可以直接通过 URL（如 http://localhost:8080/myApp/index.html）访问这些文件。Tomcat 的 Web 服务器模块会直接将其返回，无需惊动后厨。

    B. WEB-INF 目录 (私密后厨 - 绝对核心)

    这是 JavaWeb 项目中最重要、最特殊的目录。浏览器绝对不允许直接访问 WEB-INF 及其内部的任何文件，这从根本上保证了后端核心逻辑和配置文件的安全。

    • /classes 目录： 存放我们编写的 Java 业务代码编译后生成的 .class 字节码文件。Servlet 容器收到请求后，就是来这里找对应的 Java 类执行逻辑。
    • /lib 目录： 存放项目依赖的第三方 .jar 工具包（比如连接数据库的 JDBC 驱动、JSON 解析包等）。
    • web.xml 文件： 传统的“路由排班表”和“菜单”。它负责告诉 Tomcat 容器：哪个请求路径（URL）对应着 /classes 目录里的哪个具体的 Servlet 类。

    4. 深度理解：容器为何叫“容器”？

    很多人容易将计算机里的名词产生物理联想，认为它是个储物柜。实际上，容器本质上是一段写好的、一直在后台循环运行的 Java 代码。它被称为“容器”，主要是因为它是生命周期大管家和调度中心：

    1. 自动实例化（控制反转 IoC）： 开发者写好 Servlet 类后，永远不需要亲自去 new 它。容器会在服务器启动或首次收到对应请求时，自动帮你把对象创建出来。
    2. 常驻内存： 对象创建后，会被“装”在容器内部的内存中一直保存（这就是“容器”名字的核心由来），实现单例的高效复用。
    3. 路由分发： 容器内部有一套类似字典（Map）的机制，根据 web.xml 或注解的配置，将外部的网络请求精准投递给肚子里对应的那个 Servlet 对象的 service() 方法。
    4. 自动销毁： 当服务器关闭时，容器还会负责优雅地调用销毁方法，清理内存。对象的生与死，全权由容器接管。

    5. JavaWeb 技术栈演进路线

    根据架构图，Java 后端技术是逐步演进的：

    1. 底层基石： Servlet + JavaBean。直接处理 HTTP 交互，是理解底层原理的基础，但在大型项目中开发效率较低。
    2. 企业级框架（经典）： SSM 框架 (Spring + SpringMVC + MyBatis)。通过框架封装了繁琐的代码，实现了业务逻辑、控制层、数据访问层的分离，大幅提升开发效率。
    3. 现代云原生（主流）： Spring Boot + Spring Cloud。解决配置繁琐问题，提供开箱即用的特性，支持微服务架构，应对高并发、分布式场景。
