📘 Java IO 字符流与缓冲机制核心笔记

🧠 一、核心总认知（必须先建立）

💡 计算机底层只处理字节（byte），不理解字符（char）

🔁 二、字符流写入完整流程

⚙️ 三、字符流的本质

✅ 1. FileWriter 本质

👉 作用：

  • 提供字符写入接口
  • 内部完成编码转换

✅ 2. OutputStreamWriter 的作用

👉 核心职责：

✔ 将字符（char）转换为字节（byte）

👉 内部关键类：

  • StreamEncoder（重点）

🔥 四、为什么“编码过程必须有缓冲区”（重点理解）

❗ 原因1：变长编码（UTF-8）

字符 字节数
A 1
中 3
😊 4

👉 问题：

  • 一个字符 → 多个字节
  • 无法一次性确定输出大小

👉 解决：

✔ 使用缓冲区暂存编码结果

❗ 原因2：字符可能“跨边界”

👉 Java char（UTF-16）：

👉 如果分两次写：

👉 编码器在第一次：

👉 必须：

✔ 缓冲，等待完整字符再编码

❗ 原因3：编码器是“有状态的”

👉 StreamEncoder 本质：

✔ 一个 状态机（stateful encoder）

需要维护：

  • 未完成字符
  • 编码状态
  • 输出缓存

🧠 五、两种“缓冲”的本质区别（核心考点）

🟡 1. 编码缓冲（隐式）

👉 位置：

  • StreamEncoder 内部

👉 作用：

✔ 保证编码正确性

👉 特点：

  • 必须存在
  • 小规模
  • 用户不可控

🔵 2. IO缓冲（显式）

👉 代表：

  • BufferedWriter

👉 作用：

✔ 减少 IO 操作，提高性能

👉 特点：

  • 大缓冲区（默认 8KB）
  • 用户可控

🎯 对比总结

类型 编码缓冲 IO缓冲
位置 底层 上层
目的 编码正确 性能优化
是否必须 ✔ ❌
典型类 StreamEncoder BufferedWriter

⚠️ 六、Writer 中 writeBuffer 的本质

👉 作用：

❗ 仅用于方法复用的临时数组

🚫 不是：

  • ❌ 大规模缓存
  • ❌ 性能优化缓冲
  • ❌ 数据积累区

✅ 本质：

👉 避免频繁创建数组

💧 七、flush / close 的真正含义

✅ flush()

✅ close()

❗ 注意

⚠️ flush ≠ 数据真正落盘

因为：

👉 OS 也有缓存

🚀 八、完整数据流模型（终极版）

🔥 九、一句话总结（面试版）

Java 字符流通过 OutputStreamWriter 将字符编码为字节，编码过程由于变长和跨边界特性需要使用 StreamEncoder 的缓冲区保证正确性；而 BufferedWriter 提供的缓冲则用于减少 IO 调用提升性能，两者属于不同层次的机制。
