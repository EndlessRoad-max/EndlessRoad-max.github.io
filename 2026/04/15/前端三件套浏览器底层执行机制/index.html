前端三件套浏览器底层执行机制

一、浏览器加载与执行流程

1️⃣ 请求资源

浏览器向服务器请求：

  • HTML（结构）
  • CSS（样式）
  • JS（行为）

2️⃣ HTML 解析（主线）

👉 浏览器 从上到下解析 HTML

同时构建：

  • DOM 树（页面结构）

3️⃣ 解析过程中发生的事情

✅ 遇到 CSS

  • 构建 CSSOM（样式树）
  • 不阻塞 DOM（但会影响渲染）

⚠️ 遇到 JS（关键）

👉 默认行为：

  • 暂停 HTML 解析
  • 执行 JS
  • 执行完再继续解析

🔥 核心结论

HTML 解析 和 JS 执行是同步阻塞的

二、DOM 未加载问题本质

现象

本质原因

👉 DOM 还没构建出来

执行顺序

解决方案

  • script 放在 body 底部 ✅
  • DOMContentLoaded ✅
  • defer（推荐）✅
  • async（慎用）⚠️

三、BOM 与 DOM 关系

结构关系

概念区分

🌐 BOM（浏览器）

  • window
  • location
  • history

👉 不属于 DOM

🌳 DOM（页面）

  • document
  • html
  • body
  • 各种标签

👉 页面结构

🔥 关键结论

window 是浏览器对象（BOM）
document 是 DOM 入口
html 才是 DOM 根节点

四、JS 操作 DOM 的本质

👉 JS 不能直接操作页面

👉 它是通过：

👉 调用浏览器提供的接口（DOM API）

五、DOM 新增的三种方式

❌ 1. document.write（过时）

本质

👉 写入 HTML 解析流

特点

  • 写在 script 位置
  • 页面加载后调用会清空页面

✅ 2. innerHTML

本质

👉 字符串 → 解析 → DOM

✅ 3. 标准 DOM API（推荐）

本质

👉 直接操作 DOM 树

🔥 对比总结

方式 本质 推荐
document.write 写 HTML 流 ❌
innerHTML 解析字符串 ⚠️
createElement 操作 DOM ✅

六、浏览器执行本质（升维总结）

1️⃣ 两个核心引擎

  • JS 引擎（执行 JS）
  • 渲染引擎（构建 DOM）

2️⃣ 协作机制

👉 JS 通过 document 操作 DOM

3️⃣ 为什么 JS 会阻塞？

👉 因为：

JS 可以修改 DOM，必须保证执行时 DOM 是一致的

七、一句话终极总结（面试用🔥）

浏览器加载页面时会自上而下解析 HTML 构建 DOM 树，在解析过程中遇到 script 会暂停解析并执行 JS。由于 JS 可以修改 DOM，这种同步阻塞机制保证了 DOM 的一致性。如果 JS 执行时目标节点尚未构建，就会出现获取不到 DOM 的问题。window 属于 BOM，document 是 DOM 入口，而真正的根节点是 html。现代开发通过 DOM API（如 createElement、appendChild）来操作 DOM，而不是使用 document.write。
