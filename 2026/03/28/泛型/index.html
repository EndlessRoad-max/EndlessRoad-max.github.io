泛型

泛型基本概念（以集合为例子）

为什么引入泛型？

当没有泛型时面临的困境：

  1. 类型不安全（引发运行时报错）： 当我们在使用集合存入数据时，如果没有泛型，集合默认可以利用多态存入任意 Object 类型的数据。虽然一个集合往往只打算存入同一种类型（例如全存 String），但由于缺乏约束，不小心存入了其他类型（如 Integer）编译时也不会报错，直到运行时才会抛出异常。我们希望存在一种编译时的约束机制。
  2. 繁琐的向下转型： 当我们对没有泛型的集合进行遍历时，取出的元素统一返回的是 Object 对象。这时候如果需要使用该元素运行时的特有方法，就需要手动进行向下转型（强制类型转换）。我们希望通过一种机制避免手动强转，实现自动转型。

泛型的基本使用

在声明对象、实例化时通过 <引用类型> 来使用泛型机制。此时相当于类中定义的泛型变成了我们指定的具体类型。

基本使用细节

  1. 只能是引用数据类型，不能是基本数据类型。 不能写 ArrayList<int>，必须使用包装类 ArrayList<Integer>。（原因和底层的“类型擦除”有关，见后文）。
  2. 指定泛型后，可以传入该类型及其子类类型。 例如 List<Animal> list = new ArrayList<>();，你不仅可以 list.add(new Animal())，也可以 list.add(new Dog())（假设 Dog 继承自 Animal）。
  3. 如果不指定泛型，默认就是 Object 类型。 建议即使是 Object，也显式写出 ArrayList<Object>，以提高代码可读性。
  4. ⚠️ 类的泛型不能用于静态成员（变量或方法）。 静态成员是在类加载时初始化的，而类的泛型参数（如 T）是在创建对象实例化时才确定的。在静态成员初始化时，JVM 根本不知道 T 是什么，因此会编译报错。 (注：如果静态方法需要使用泛型，必须自己独立声明泛型，成为“静态泛型方法”。)

如何自定义泛型类？

我们可以在自己编写的类、接口中引入“待定类型”，这就是自定义泛型。

语法格式： class 类名<泛型标识符> { ... } 常见的泛型标识符约定俗成：T (Type), E (Element), K (Key), V (Value)。

底层原理（它在底层到底是怎样处理的）

泛型的本质是源代码（编译器）层面的语法糖。在 Java 的底层运行环境中（JVM 虚拟机内存里），根本不存在泛型这个概念。

Java 处理泛型的核心机制叫做 “类型擦除（Type Erasure）”。整个过程分为以下三个关键阶段：

阶段一：编译期的“严格安检”（类型检查）

当编译器（如 javac）读取源代码时，它会记录你指定的泛型信息（例如 ArrayList<String>）。 当代码中调用 list.add(123) 时，编译器会根据记录的泛型 String 进行比对。发现 123 是整数，与 String 不符，立刻抛出编译错误，拒绝生成字节码文件。这就是泛型起到的“约束”作用。

阶段二：卸下伪装的“类型擦除”

一旦所有的代码都通过了编译器的类型检查，编译器在生成 .class 字节码文件之前，会执行“擦除”操作：

  1. 把所有的泛型标识符（如 <T>、<String>）全部抹掉。
  2. 将类中使用了泛型的地方，统一替换为 Object（如果泛型有限制，比如 <T extends Number>，则替换为父类 Number）。

阶段三：编译器悄悄补全的“向下转型”

既然底层全变成了 Object，为什么我们取数据时不需要手动写向下转型了呢？ 因为编译器在执行类型擦除的同时，会在你调用 get() 等获取数据的方法处，自动帮你生成一条强制类型转换的字节码指令。

为什么 Java 要设计成“类型擦除”？

主要原因是历史包袱（向下兼容）。Java 在 JDK 1.5 才引入泛型，为了保证新加了泛型的代码编译后，依然能在老版本的 JVM 上运行，且能和以前没有写泛型的老代码完美混用，Java 团队妥协采用了这种只在编译器做检查，底层维持 Object 原样的方案。这也解释了为什么基本数据类型（如 int）不能作为泛型参数，因为它无法强转为 Object。
